## Auto-compact context-spike bug: root cause + fix

User reported auto-compact "very minorly compacts" then context/usage
immediately shoots back up, and fresh sessions/`/clear` also "fill the
context a lot." Root-caused via `superpowers:systematic-debugging`
Phase 1-3, and live-corroborated: a real compact fired mid-investigation in
this very session, directly reproducing the symptom while under
observation.

**Cause:** two enabled plugins — `ponytail` and `superpowers` — register a
`SessionStart` hook with a matcher including `compact`, so Claude Code
re-fires them not just on real startup/`/clear` but after every auto-compact
too. Each firing re-injects fixed content: ponytail's full-mode ruleset
(~1,325 tokens) and superpowers' entire `using-superpowers` skill text
(~765 tokens) — ~2,100 tokens re-added immediately after a compaction that
was supposed to shrink context. Separately observed but not fixable: Claude
Code's own compaction routine independently replays full prior
skill-invocation text and tool-call/file-read results (~7-9K tokens) into
the post-compact context — internal CLI behavior with no accessible source.

**Fix:** patched both plugins' cached hook manifests directly (mechanical
edit via implementer subagent, verified by direct read after):
- `ponytail/4.8.3/hooks/claude-codex-hooks.json` — removed the
  `SessionStart` block entirely. Redundant even before the compact-matcher
  issue: the user's own `~/.claude/CLAUDE.md` Operating Mode section
  already keeps ponytail off in the main session and activates it only via
  explicit subagent-prompt text, so the main-session hook injection had
  zero functional value. `SubagentStart`/`UserPromptSubmit` hooks (which do
  the real work) left untouched.
- `superpowers/6.1.0/hooks/hooks.json` — narrowed `SessionStart` matcher
  from `startup|clear|compact` to `startup` only.

Not committed to this repo — the patched files live in
`~/.claude/plugins/cache/`, outside version control, and will be silently
reverted by a future plugin update. Noted in project memory
(`project_hook_matcher_patch_fragility`) so a future session checks plugin
versions first if the symptom reappears.

## Proxy Rework & Prompt Optimization

Reworked Winnow's trimming design to achieve actual token savings while keeping prompt caching 100% operational:
- **Root Caused Cache Misses**: Query-dependent BM25/embedding scoring on every turn caused older messages to flip-flop, busting the prompt cache (writes cost 1.25x). Freezing everything before the last `cache_control` marker stopped cache-busting but saved exactly 0% tokens in practice because Claude Code places the cache marker at the very end of the history.
- **Stable Moving-Window Trimming (SMWT)**: Trims older `head` messages (outside `keep_last_turns` window, default 2) deterministically: `tool_result` blocks (95%+ of context size) are stubbed to `[winnow: tool result omitted]`, and young tool results are truncated (JSON to 20 items, text to 8000 characters). This query-independent, deterministic moving window is perfectly prefix-preserving and cache-friendly, allowing 100% cache hits on subsequent turns while dropping prompt size by 90%+.
- **System Prompt Minification**: Added automatic regex-based minification of the `system` prompt in the proxy, dynamically detecting and stripping ponytail rules (~1,325 tokens) and superpowers bootstrap instructions (~765 tokens) in the main session. This saves a flat 2,100+ tokens on every turn.
- **CLAUDE.md Minification**: Compressed rules in the active `~/.claude/CLAUDE.md` and shareable `dotfiles/claude/CLAUDE.md` into highly concise instructions, saving ~1,300 tokens on every request.
- **Test Integrity**: Updated `winnow/tests/` to align with the new design; all 51 tests pass.

## Four-Tier Task Complexity Hierarchy & Fallback Mechanics

- **Four-Tier Execution Flow**: Formally codified the four-tier model execution hierarchy inside `CLAUDE.md`:
  - **Tier 0 (Claude Opus 4.8)**: High-risk/critical tasks—extreme debugging loops where Sonnet is stuck, core safety audits, or final architecture verification.
  - **Tier 1 (Claude Sonnet 5)**: Standard workhorse—complex logic, multi-file edits, codebase-wide searches, planning, and signal-processing math.
  - **Tier 2 (Claude Haiku 3.5)**: Medium-complexity subagent tasks (single-file scripts, refactorings) and fallback for Tier 3.
  - **Tier 3 (Local oMLX)**: Genuinely low-complexity/single-shot tasks (explanations, docstrings, regex, summarizations).
- **Graceful Fallback Mode**: When oMLX is down or on battery, the flow automatically collapses to Tier 1 (Sonnet 5) and Tier 2 (Haiku).

## Winnow Advanced Proxy Capabilities

- **Dollar-Cost Sentinel (Budget Safety Cap)**: Upgraded the raw token cap to a model-aware dollar-cost cap (`WINNOW_MAX_SESSION_COST` - default $0.30). It calculates cost dynamically depending on whether Opus, Sonnet, or Haiku is requested, protecting your budget accurately.
- **Transparent Local oMLX Routing Intercept**: Winnow now intercepts simple prefix questions (`explain:`, `doc:`, `test:`, `regex:`, `commit:`) or simple query prompts, processes them locally via the oMLX server, and streams the answer back to the Claude Code client natively. This completely avoids making a call to Anthropic's API, keeping the experience 100% inside your standard Claude Code terminal.
- **Smart Git Diff Compression**: Automatically strips out massive lockfile changes (e.g. `package-lock.json`, `yarn.lock`) from git diff outputs passing through tool results, saving thousands of wasted context tokens.
- **Preloaded File Stripping**: Winnow now intercepts turn-zero messages and automatically strips out the raw text contents of all files pre-loaded from open editor tabs. It replaces them with a lightweight reference placeholder, prompting the model to use the `view_file` tool to read them on-demand if actually needed. This prevents massive startup context bloat.
- **Git Hygiene Startup Guidelines**: Added a rule in `CLAUDE.md` to establish clean workspace habits. Since Claude Code automatically loads all modified files and uncommitted diffs on startup (which can consume 110k+ tokens instantly), committing or stashing files before starting a session ensures the startup context remains under 5,000 tokens.
