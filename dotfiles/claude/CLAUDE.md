## Retrieval & Tool Discipline
- Prefer Serena's symbol tools (find_symbol, find_referencing_symbols, get_symbols_overview) over reading whole files for code navigation. Fall back to `rg` for text search, `ast-grep` (CLI or MCP) for structural/pattern matches, `fd` for filename search.
- Before assuming a library/API's shape, resolve it via Context7 (resolve-library-id → query-docs) instead of relying on memory.
- Use the arxiv-mcp-server tools for paper search/reading. There is no vetted Semantic Scholar/OpenAlex/Papers-with-Code MCP worth adding (see claude-orchestra/docs/DECISIONS.md) — fall back to WebSearch/Firecrawl for other literature.
- Model routing happens through subagent delegation, not by switching the main session's own model (that's not possible mid-conversation). When a piece of work is genuinely low-thought — formatting, markdown/prose cleanup, grammar fixes, straightforward summaries, mechanical boilerplate, simple/obvious lookups — actively spawn a subagent via the Task/Agent tool with `model: haiku` instead of doing it inline or defaulting to sonnet/opus. Reserve sonnet/opus subagents (and the main session itself) for implementation, debugging, architecture, research synthesis, and anything requiring real judgment. Err toward delegating when a subtask is clearly separable and low-effort; when in doubt, keep it in the main session.

## Operating Mode
- Caveman ultra intensity is active every response, session-wide (main session and subagents alike) — not just when explicitly requested. In addition to caveman's own existing Auto-Clarity exceptions (security warnings, irreversible-action confirmations, multi-step sequences, user confusion — always exempt from compression), also treat reporting on completed implementation work as exempt: when summarizing what a subagent built or changed, use clear explanatory prose, not ultra-compressed fragments.
- Apply `andrej-karpathy-skills:karpathy-guidelines` (avoid overcomplication, make surgical changes, surface assumptions, define verifiable success criteria) on every response, not only code-focused ones — treat it as always-active, the same way ponytail declares itself active every response.
- Ponytail's lazy-code-minimalism principles do not apply to the main session's own reasoning, planning, or brainstorming — the main session runs in ponytail's normal/off mode at all times. Ponytail is invoked explicitly only inside the prompt of an implementer subagent, once real implementation begins (see next point). This is a deliberate override of ponytail's plugin-level "active every response" default.
- Once brainstorming/planning has produced a concrete plan, decompose it into subgoals, identify which are independent (dispatch via `superpowers:dispatching-parallel-agents`) versus sequential, and execute via `superpowers:subagent-driven-development`'s per-task implementer → reviewer loop. The main session does not edit files or run implementation/build commands directly once a plan exists — that work is always delegated to implementer subagents; the main session decomposes, dispatches, monitors subagent status, and reports results (in explanatory form per the caveman exception above). Scale the process to the task: a genuine multi-step plan gets the full flow (worktree, per-task review, final review); a trivial single-file/single-line mechanical fix still gets delegated to one implementer subagent but skips the full worktree/multi-review ceremony. This is a default workflow preference, not a hard technical lock — if a subagent is genuinely blocked in a way that would stall the user's actual request, act directly rather than failing the request outright. Claude Code's existing functionality and tool availability are never restricted by this rule.
